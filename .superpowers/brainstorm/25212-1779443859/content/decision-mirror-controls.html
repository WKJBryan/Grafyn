Where should model choice and thinking visibility live?

This compares three UI shapes for Decision Mirror / Twin prompts. Click the one that feels closest.

A. One Prompt Dialog
Decision
Should I choose X, Y, or something else?
Options
A / B / C entered by user
Models
Same multi-model dropdown as + Add Models
Thinking
Effort slider + “show reasoning summary” toggle

Put everything before Run

Best for explicit control. The user chooses all models up front, including local models, then all responses run together.

B. Lean Prompt, Rich Response Cards
Prompt Dialog
Question + options + compact model preset row
Response Node
Answer
Why this option / assumptions / uncertainty
Thinking used: High · Sources: 4 · Outside-options considered: Yes

Show reasoning after the answer

Best for readability. The dialog stays manageable; each model card exposes a structured reasoning summary and option coverage.

C. Separate Compare Drawer
Prompt Tile
Decision + selected options
Compare Drawer
Model dropdown / presets / local models
Reasoning effort per run
Run same prompt on additional models

Keep comparison as an add-on workflow

Best for less clutter. Start with one configured model, then use the drawer to add simultaneous comparisons and reasoning visibility.

Current diagnosis from the code

The options field is currently treated as decision metadata and copied into the prompt. It asks the model to compare those options, but it does not explicitly ask for “outside option / none of these / reframed option.” That is why the answers tend not to jump outside the list.

The “Thinking level” control already exists, but it sends only an OpenRouter reasoning effort setting. The backend stream parser currently keeps normal answer content only, so there is no separate visible thinking or reasoning-summary surface in the response node.
